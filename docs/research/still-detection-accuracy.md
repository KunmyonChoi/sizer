# 정지 구간 감지 정확도 개선 — 리서치

날짜: 2026-07-22
대상: 트리밍(잘라내기)·빨리감기의 정지/저모션 감지 정확도

## 1. 현재 구현

- `FreezeDetector.detectFreezes`: **단일 패스** `freezedetect=n=<noiseDb>:d=<minStillDuration>`.
  freezedetect는 **프레임 전체(모든 성분)의 평균 절대 차(mean absolute difference)를 노이즈 플로어와 비교**한다(공식 문서 확인). 기본 -60dB(=0.001).
- 후처리(`SegmentPlanner`): 여집합(keep) → `mergeShortGaps` → `dropShortSegments` → `pad` → 안전장치.
- `FreezeDetector.detectSceneChanges`(`select='gt(scene,0.4)'`)가 있으나 **`ConversionEngine`이 `sceneChanges: []`를 넘겨 현재 비활성**(장면 전환 병합 가드가 꺼져 있음).
- 민감도 프리셋은 `(noiseDb, minStill, mergeGapMax)`만 조절 → **본질은 "고정 dB 임계값"**.

## 2. 실험 (합성 클립, 실제 ffmpeg 빌드)

18초 클립: [0–4] 움직임 · **[4–9] 정지** · [9–11] 움직임 · **[11–16] 정적+작은 커서(18px 이동)** · [16–18] 움직임.

### 실험 1 — 깨끗한 클립
- freezedetect(-40/-50/-60dB)·mpdecimate **모두 [4,9]·[11,16] 정확 감지**.
- 즉, **"작은 커서/스피너가 freezedetect를 무력화한다"는 통념은 과장** — 18px 요소의 전체 프레임 평균차는 사실상 0(mafd≈0.000)이라 문제 없음.

### 실험 2 — **노이즈 있는 정지**(압축/센서 노이즈 모사: `noise=alls=18:allf=t`)
정답 정지: [4,9] 노이즈-정지, [11,16] 커서-정지.

| 방법 | [4,9] 노이즈-정지 | [11,16] 커서-정지 |
|------|------------------|-------------------|
| A. freezedetect −50dB (현재) | ❌ **놓침** | ✅ |
| B. freezedetect −60dB (더 엄격) | ❌ 놓침 | ✅ |
| C. `hqdn3d=4:3:6:4` + freezedetect | ❌ 놓침 | ✅ |
| D. **`scale=iw/4:ih/4,boxblur=2` + freezedetect** | ✅ **감지** | ✅ |
| E. mpdecimate | ❌ 놓침 | ✅ |

- scdet `lavfi.scd.mafd` 초당 평균: 노이즈-정지 ≈ **3.9–4.7**, 깨끗-정지 ≈ **0.0**, 실제 움직임 ≈ **0.8–1.7**.
  → 노이즈가 정지 신호를 움직임보다 더 높게 끌어올려 **고정 임계값으로는 분리 불가**.

### 핵심 결론
1. **실제 실패 모드는 "작은 커서"가 아니라 "노이즈 플로어"**다. 화면 전체에 깔린 압축/센서 노이즈가 정지 프레임의 평균차를 임계값 위로 올려 **정지 구간을 통째로 놓친다**. 임계값을 **더 엄격(-60dB)** 히 해도 악화될 뿐(감지하려면 오히려 -30dB처럼 느슨하게 해야 하나 그러면 실제 저모션에서 오검출).
2. **mpdecimate도 균일 노이즈에는 취약**(블록 SAD가 걸림). 작은 국소 움직임엔 강하지만 노이즈 해법은 아님.
3. **가장 효과적인 개선은 감지 전 다운스케일+블러 프리필터**다. 픽셀 평균화로 노이즈 플로어가 무너져, **임계값을 그대로 두고도** freezedetect가 노이즈-정지를 정확히 잡았다(방법 D). `hqdn3d` 기본값보다 강력하고, **저해상도 프록시라 감지 속도도 빨라진다**(160×90 분석 = 픽셀 ~144× 감소).
4. `lavfi.scd.mafd`는 **프레임마다** 출력되어(실측 확인) 코드에서 직접 임계·히스테리시스에 쓸 수 있는 원시 정지 신호다.

## 2b. 실제 샘플 검증 (`미장센빌더.mov`, 2032×1162 60fps, 45.3분)

- 전체 파일: 현재(-58dB d3) 기준 **13구간·총 1894.6s 정지(약 70%)** — 이 녹화는 대부분 정적인 빌더 화면. **현재 방식이 이미 잘 잡고 있음.**
- 45분 풀 패스는 소프트웨어 디코드로 ~2.5분/패스 → 느림. **다운스케일 프록시는 디코드가 병목이라 실측 ~10%만 빠름**(16.0s→14.4s/241s 슬라이스). 즉 프록시의 주 가치는 "속도"가 아니라 **노이즈 강건성 기반**(블러가 노이즈 floor를 낮춰 적응형 임계값이 동작)이고, 속도는 부수적. (큰 속도 개선을 원하면 `-hwaccel videotoolbox` 디코드가 별개 지렛대.)
- 4분 슬라이스(깨끗): 현재 A와 프록시 C가 **완전히 동일**(3구간 11.0s) → **깨끗한 화면 녹화에선 무회귀(neutral)**.
- **슬라이스에 노이즈 추가 후 레시피 스윕**(정답 ≈ 11.0s):

  | 레시피 | 결과 |
  |--------|------|
  | scale¼+blur2, **-58**dB | 0s ❌ (너무 엄격, 놓침) |
  | hqdn3d(강), -58dB | 0s ❌ |
  | **scale¼+blur2, -50dB** | **7.7s** ✅ (정답 근접, 약간 보수) |
  | scale⅛+blur4, -50dB | 7.8s ✅ |
  | scale⅛+blur3, -58dB | **116s** ⚠️ (과검출 — 움직임을 정지로) |
  | hqdn3d+scale¼, -58dB | 109s ⚠️ 과검출 |
  | scale¼+blur2, **-45**dB | **120s** ⚠️ 과검출 |

### 검증으로 수정된 결론
- **초기(합성만) 결론 "다운스케일+블러가 큰 정확도 이득"은 과대평가였다.** 실제+노이즈로 보면 **좁은 안전 구간이 있는 보정 노브**다: 과한 블러(1/8)나 느슨한 고정 임계값(-45)은 **움직임을 정지로 오검출(116~120s)** → 실제 콘텐츠를 잘못 자르거나 배속함(치명적).
- 노이즈 콘텐츠의 안전 스윗스팟: **약한 프록시(scale¼, blur2) + 중간 임계값(-50dB)** → 7.7/11s 회복, 과검출 없음.
- **고정 dB의 한계가 핵심** — 콘텐츠마다 안전 임계값이 다르다(깨끗=-58 OK, 노이즈=-50 필요, -45는 과함). 따라서 **가장 원칙적인 해법은 적응형 임계값**이고, 다운스케일 프록시는 "속도 + 약한 강건성" 보조 수단으로 자리매김.

## 3. 권고 (우선순위, 실제 검증 반영)

### #1 — 감지 전 다운스케일+블러 프리필터 (최우선, 저비용·저위험)
`FreezeDetector.detectFreezes`의 vf를
```
scale=iw/4:ih/4,boxblur=2,freezedetect=n=<noiseDb>:d=<minStill>
```
로 변경. 노이즈-정지 놓침을 해결하고 감지 속도도 향상. (fast-forward의 freeze 감지에도 동일 적용 — 같은 함수 재사용.)
- 튜닝: 다운스케일 배수(1/4)와 `boxblur` 반경은 콘텐츠에 따라 조정. 텍스트가 많은 화면 녹화는 1/4가 안전, 더 강한 노이즈는 `scale=iw/6:-2` 또는 `boxblur=3`.
- 대안/보강: `hqdn3d`는 기본값이 약했음 → 쓰려면 강하게(`hqdn3d=8:6:8:6`) 또는 `smartblur`.

### #2 — 장면 전환 가드 활성화 (현재 비활성, 정확도 무료 이득)
`ConversionEngine`이 `detectSceneChanges` 결과를 `SegmentPlanner.plan(..., sceneChanges:)`에 전달하도록 연결. `select='gt(scene,0.4)'`는 신형 `scdet`로 교체 권장(더 안정적 점수/메타데이터). 프록시(다운스케일) 위에서 돌리면 비용도 작다.

### #3 — 적응형 임계값(자동 보정)
클립의 노이즈 플로어를 표본화(예: `scdet` mafd 중앙값)해 **임계값을 상대적으로** 설정. 고정 dB의 콘텐츠별 오차를 제거. 구현 부담이 있으나 다양한 소스(폰 영상 vs 깨끗한 화면 녹화)에 견고.

### #4 — 히스테리시스 + 참조-프레임 비교 (선택)
- 히스테리시스: 진입은 엄격, 이탈은 느슨 → 경계 흔들림(마이크로컷) 감소.
- 참조(고정) 프레임 비교: 인접 프레임이 아니라 정지 후보의 **첫 프레임**과 비교 → 느린 드리프트(페이드·느린 진행바) 포착. ffmpeg 네이티브로는 어려워 `scdet` mafd 스트림을 코드에서 처리하는 방식이 현실적.

## 4. 제안 실행안(최소 변경, 큰 효과)
1. `FreezeDetector.detectFreezes` vf에 `scale=iw/4:ih/4,boxblur=2,` 프리픽스 추가(#1).
2. `detectSceneChanges`를 `scdet` 기반으로 바꾸고 프록시에서 실행, `ConversionEngine`에서 `SegmentPlanner.plan`에 전달(#2).
3. (후속) 적응형 임계값·히스테리시스는 별도 이터레이션(#3/#4).

## 5. 검증 유의
- 위 수치는 **합성 클립** 기준. 실제 화면 녹화·폰 캡처 샘플로 재검증 필요.
- 다운스케일 배수/블러 반경은 오검출↔미검출 트레이드오프가 있으니, 프리셋(공격적/균형/보수적)과 함께 튜닝하고 기존 `FastForwardTests`류 통합 테스트로 고정.

## 6. 후속 실측 & 메모 (2026-07-22, v1.5.0 이후)

### 잘라내기 vs 빨리감기 — 탐지 기준은 공유
- `ConversionEngine`이 `switch` **이전에** `detectFreezes(options: opts)`를 호출 → 민감도/`noiseDb`, 최소 정지 길이, 프록시 감지, 적응형 임계값은 **두 모드에 동일 적용**.
- 차이(후처리): 잘라내기 = 컷 병합·최소 유지·패딩·안전장치 + **장면 가드**. 빨리감기 = `ffMinDuration`(기본 2.0s) 이상 구간만 배속, 장면 가드·병합·패딩 없음.

### `-hwaccel videotoolbox` — 도입 안 함 (실측, 241s 슬라이스)
| 방식 | 감지 시간 |
|------|-----------|
| SW + 풀해상도 | 14.6s |
| **SW + 프록시(현재)** | **11.0s** |
| HW + 풀해상도 | 38.7s (2.6× 느림) |
| HW + 프록시 | 27.0s (2.5× 느림) |
- 분석 패스(-f null)는 소프트웨어 필터 때문에 프레임을 GPU→CPU로 되내려야(hwdownload) 해서 전송·동기화 오버헤드가 SW 디코드보다 큼 → **HW가 오히려 2.5× 느림**.
- 게다가 HW 디코드는 색공간 처리 차이로 **freezedetect 결과가 달라짐**(SW 4구간 vs HW 3구간) → 정확도 리스크. **속도·정확도 모두 이유로 미도입.**
- 프록시 실측 속도 개선은 풀해상도 대비 ~10~25%(머신 부하 의존). 감지는 디코드 병목이라 정확도 유지 시 여기가 실질 상한.

## 참고

- mpdecimate(8×8 블록 단위 비교, `hi`는 블록 하나 기준): FFmpeg 필터 문서 / 해설.
